# Installation Guide

This guide explains how to implement the `ai-artifacts` pattern in a repository. The goal is to manage AI instructions, skills, agents and tool configuration as versioned, auditable artifacts instead of hand-edited local copies.

## Target Setup

A consuming repository should keep source artifacts, generated outputs and validation separate:

| Area | Path | Purpose |
|---|---|---|
| Artifact playbook | `.ai-artifacts/artifacts.yml` | Declares upstream packages and generated artifacts. |
| Local sources | `.ai-artifacts/files/` | Repository-owned instructions, skills, agents and config source files. |
| Local overlays | `.ai-artifacts/overlays/` | Small repository-specific additions appended to upstream artifacts. |
| Lock metadata | `.ai-artifacts/lock.yml` | Resolved upstream commits and generated content hashes. |
| Reports | `.ai-artifacts/reports/` | Drift, risk and update summaries generated by validation. |
| Generated outputs | `AGENTS.md`, `.opencode/*`, `.github/*` | Files consumed by agents and tools. |

The important rule is simple: generated outputs are not the design surface. Change the source under `.ai-artifacts/`, then regenerate.

## Install The Package

Inside this incubation repository, the package lives in `packages/ai-artifacts/` and is wired through root npm scripts. A future external consumer should install the package normally once it is published or referenced as a dependency.

For this repository, bootstrap with:

```bash
npm install
npm run validate:ai-artifacts
```

The root `postinstall` installs packaged automation into repository locations, currently the GitHub workflow and JSON schema.

## Create The Playbook

Start from `packages/ai-artifacts/templates/artifacts.yml` and adapt it to the target repository.

Minimum structure:

```yaml
# yaml-language-server: $schema=./schemas/artifacts.schema.json
version: 1

packages:
  hve-core:
    type: git
    repo: https://github.com/microsoft/hve-core.git
    version: hve-core-v3.2.2

artifacts:
  - id: repo-instructions
    kind: agent
    target: AGENTS.md
    steps:
      - copy:
          from: local:AGENTS.md
          to: AGENTS.md
```

Use `copy` for local files that should be preserved exactly. Use `render` for upstream artifacts that need substitutions and overlays.

## Recommended Generated Outputs

For an opencode-based repository, generate these outputs:

| Output | Why |
|---|---|
| `AGENTS.md` | Shared repository instructions for coding agents. |
| `CLAUDE.md` | Claude-compatible copy when needed. |
| `.opencode/opencode.json` | Project-level instructions, skills path, permissions and output limits. |
| `.opencode/agent/*.md` | Repository-specific subagents. |
| `.opencode/skills/*/SKILL.md` | Reusable task skills and workflow triggers. |

Keep opencode config minimal. Prefer files for agents and skills instead of large inline config blocks.

## Agent And Skill Practices

Good skills are narrow and triggerable. Each `SKILL.md` should have frontmatter with a `name` and a concrete `description`. The description should say when to use the skill, not just what it is.

Good agents have a role, permissions and validation expectations. Use read-only permissions for research and review agents. Allow edits only for implementation, prompt-building or documentation agents that genuinely need to write files.

Recommended baseline agents:

| Agent | Role | Edit Permission |
|---|---|---|
| `task-researcher` | Finds facts, patterns, alternatives and constraints. | deny |
| `task-planner` | Builds small implementation plans. | deny |
| `task-implementor` | Makes scoped code/docs/artifact changes. | ask |
| `task-reviewer` | Reviews for bugs, drift and missing validation. | deny |
| `prompt-builder` | Maintains prompt, skill, agent and instruction artifacts. | ask |
| `doc-ops` | Keeps documentation accurate and useful. | ask |

For long-form whitepaper work, prefer a composed workflow rather than one oversized skill: research, outline, edit, review and publish.

## Validation Workflow

Use these commands from the repository root:

```bash
npm run ai-artifacts:fetch
npm run ai-artifacts:sync
npm run ai-artifacts:sync -- --check
npm run ai-artifacts:doctor
npm run validate:ai-artifacts
```

Run `sync` after changing `.ai-artifacts/files/`, `.ai-artifacts/overlays/` or `.ai-artifacts/artifacts.yml`. Run `validate` before opening a PR.

Use `doctor` when setting up a developer machine or debugging agent integration. It checks Node, git, required artifact files, installed workflow/schema files and common opencode outputs. It also attempts to identify whether the terminal appears to run under opencode, Claude Code, Kiro, Cursor, VS Code or GitHub Copilot. This detection is heuristic: different tools expose different environment variables, and some expose none.

CI should check at least:

- Upstream packages can be fetched.
- Generated artifacts are up to date.
- Packaged automation is installed.
- Drift, risk and summary reports can be generated.
- Package tests pass when the package changes.

## Git Hygiene

Commit source and generated outputs together. This makes PRs reviewable: reviewers can inspect the intended source change and the exact generated result.

Track:

- `.ai-artifacts/artifacts.yml`
- `.ai-artifacts/files/**`
- `.ai-artifacts/overlays/**`
- `.ai-artifacts/lock.yml`
- Generated outputs consumed by tools

Usually ignore:

- `.ai-artifacts/vendor/`
- `.ai-artifacts/reports/`
- `node_modules/`

## Adoption Guidance

Start small. Add repository instructions first, then one or two high-value skills, then agents once the repeated workflows are clear. Avoid creating a central imposed framework before teams have learned what actually helps.

Standards should become de facto standards because they prove useful. Keep local guidance concrete: commands, paths, validation rules, ownership and examples from the repository.

After changing opencode config, agents or skills, restart opencode. Running sessions keep the previously loaded configuration.
